# 测速与反代测试逻辑分析

## 一、普通测速逻辑

### 1. 触发入口

- CLI：`cmd/bestsub/main.go` 中 `-run` 调用 `app.RunOnce(ctx, cfg, push)`。
- Web：`POST /api/probe/run` 调用 `app.RunOnceMode(ctx, cfg, push, mode)`。
- 默认模式是 `quick`；Web/内部可传 `stable`。

### 2. 主流程

核心在 `internal/app/app.go`：

1. 读取并应用配置。
2. 如果配置了 `worker.password`，先登录 Worker，避免测完后推送才发现密码错误。
3. 如果开启 `probe.preflight.enabled`，执行预检。
4. 调用 `source.Load` 加载候选 IP。
5. 调用 `probe.Run` 并发测速。
6. `stable` 模式会对首轮成功结果追加 5 轮复测。
7. 调用 `probe.Keep` 筛选最终结果。
8. 调用 `probe.FormatADD` 写入 `ADD.txt`。
9. 如果请求推送，则调用 Worker `/admin/ADD.txt`。

### 3. 候选 IP 加载

实现位置：`internal/source/source.go`。

- 来源支持：
  - `file`：读本地文件，如 `seeds.txt`。
  - `remote` / `remote_cidr`：HTTP 拉取远程内容。
- 每个来源会拆成 token，支持：
  - 单 IP：按配置端口展开为多个候选。
  - `IP:port`：直接作为一个候选。
  - CIDR：后续按预算随机生成 IP。
- `candidate_limit` 决定候选上限。
- 本地明确写死的 IP 会优先保留，CIDR 随机生成只填补剩余预算。
- 最后按 `IP:port` 去重。

### 4. 单个节点测速

实现位置：`internal/probe/probe.go` 的 `One`。

对每个候选执行真实网络请求：

1. TCP 连接：直连 `候选IP:端口`。
2. TLS 握手：`ServerName` 使用 `probe.target.sni`。
3. HTTP 请求：手写 HTTP/1.1 请求。
   - 请求路径来自 `probe.target.url` 的 path/query。
   - `Host` 使用 `probe.target.host`。
   - 方法默认 `HEAD`。
4. 读取 HTTP 响应。
5. 记录耗时：
   - `tcp_ms`：TCP 建连耗时。
   - `tls_ms`：TLS 握手耗时。
   - `http_ms`：写入请求到读到响应头耗时。
   - `total_ms`：总耗时。
6. 响应状态码命中 `probe.target.expected_status` 才算成功。
7. 从 `Cf-Ray` 响应头解析 Cloudflare colo，例如 `xxx-NRT`。
8. 根据内置 colo 映射得到国家码和国家名。

注意：这里测的是“入口 IP 到 Worker 目标域名”的可用性和延迟，不下载大文件，不计算带宽。

### 5. 并发、排序和筛选

`probe.Run` 使用 worker pool 并发执行，默认并发来自 `probe.concurrency`。

排序规则：

1. 成功结果排前。
2. `total_ms` 越低越前。
3. `http_ms` 越低越前。
4. 来源权重 `source_weight` 越高越前。
5. 端口号越小越前。

最终保留逻辑 `probe.Keep`：

- 只保留成功结果。
- 如果配置了 `probe.countries`，只保留匹配国家。
- 每个 IPv4 `/24` 默认最多保留 `per_cidr24_limit` 个，IPv6 按 `/64` 去重限制。
- 最多保留 `probe.keep` 个。

### 6. quick 与 stable 模式

`quick` 模式：

- 如果 `candidate_limit > 600`，压到 600。
- 如果 `timeout_ms > 2500`，压到 2500ms。

`stable` 模式：

- `candidate_limit` 至少 1800。
- `timeout_ms` 至少 4000ms。
- 并发最多 100。
- 首轮成功结果取前 100 个，再额外复测 5 轮。
- 合并时更看重成功率、成功次数、延迟波动和平均延迟。

### 7. 预检逻辑

实现位置：`internal/preflight/preflight.go`。

- 检查系统代理。
- 检查 IPv6 连通性。
- 抽样测速，计算成功结果的总延迟中位数。
- 如果中位延迟低于 `low_latency_threshold_ms`，认为可能走了本机代理、透明代理或 TUN；配置 `block_on_low_latency` 时会阻止正式测速。

当前 `configs/config.yaml` 里预检是关闭的：

```yaml
probe:
  preflight:
    enabled: false
    disable_sample_probe: true
```

## 二、反代测试 / PROXYIP 优选逻辑

### 1. 触发入口

- Web：点击“优选反代 IP”，请求 `POST /api/proxyip/fetch?country=XX`。
- 后端：`internal/web/server.go` 调用 `app.FetchProxyIPOnly(ctx, cfg)`。
- 该流程只优选反代 IP，不会触发普通入口 IP 测速。

### 2. 主流程

核心在 `internal/proxyip/proxyip.go` 的 `FetchAndCheck`：

1. 检查 `clash.proxyip_auto.enabled`。
2. 根据国家码筛选反代候选。
3. 可选本地 GeoIP 过滤。
4. 如果启用 `worker_verify`，通过 Worker 实测反代出口。
5. 如果未启用 `worker_verify`，走第三方 check API 检测延迟。
6. 按响应时间升序排序，取前 `limit` 个。
7. 返回逗号拼接的 PROXYIP 字符串，并保存到 `PROXYIP.txt`。

### 3. 候选来源与国家过滤

默认来源：

```yaml
clash:
  proxyip_auto:
    source_url: https://zip.cm.edu.kg/all.txt
```

程序下载 `all.txt` 后调用 `parseAllTXT` 解析。该列表一般按国家分组，程序只保留配置国家下的 IP 或 `IP:port`。

当前配置：

```yaml
proxyip_auto:
  enabled: true
  country: JP
  limit: 3
  max_candidates: 50
  concurrency: 1
  require_geoip_match: true
  geoip_db_path: GeoLite2-Country.mmdb
```

因此当前会优选日本反代，最多抽样 50 个候选，最终保留 3 个。

### 4. 本地 GeoIP 过滤

如果 `require_geoip_match: true`：

- 在网络测试前先用 `GeoLite2-Country.mmdb` 检查候选 IP 注册地。
- 只有 GeoIP 国家与目标国家一致才进入后续测试。

这一步能减少无效测试数量，但也可能误杀 CDN、Anycast 或数据库不准的 IP。

### 5. Worker 真实出口验证

当前配置启用了：

```yaml
worker_verify:
  enabled: true
  url: https://cloudflare.com/cdn-cgi/trace
  max_checks: 10
```

执行逻辑：

1. 先登录 Worker，复用带 cookie 的 HTTP client。
2. 从候选中随机抽样最多 `max_candidates` 个。
3. 并发请求 Worker：

```text
GET {worker.base_url}/admin/proxyip-test?proxyip={候选反代IP}&url={测试URL}
```

4. Worker 端应使用该 `proxyip` 发起到测试 URL 的请求，并返回 JSON。
5. 本地校验返回：
   - `success == true`
   - 返回的 `country` 等于目标国家
   - 返回的出口 `ip` 非空且格式正确
   - 如果有 GeoIP DB，再校验出口 IP 注册地等于目标国家
6. 通过后记录候选反代 IP 和 `responseTime`。
7. 达到 `limit` 后提前取消剩余测试。

也就是说，`worker_verify` 测的是“Worker 使用该 PROXYIP 后，真实访问外部 URL 的出口国家和耗时”，比第三方 check API 更贴近日常使用场景。

### 6. 第三方 check API 逻辑

如果没有启用 `worker_verify`，则调用：

```yaml
check_api: https://api.090227.xyz/check?proxyip=%s
```

每个候选会请求格式化后的 check API，解析 JSON：

```json
{
  "candidate": "...",
  "success": true,
  "responseTime": 123
}
```

只有 `success: true` 的候选会进入排序。该方式依赖第三方接口，不一定等价于你的 Worker 实际出口。

## 三、反代结果如何使用

本次整理后已移除本地 Clash 配置生成功能，反代链路只保留 `PROXYIP.txt`：

- 反代优选结果保存到本地 `PROXYIP.txt`。
- Web 界面可快速复制筛选结果。
- 点击“同步到 CF”会调用 `POST /api/worker/proxyip`。
- 后端登录 Worker 后把结果推送到 Worker 的 `/admin/PROXYIP.txt`。

配置里仍保留 `clash.proxyip_auto` 这一层，仅用于兼容旧配置文件；它现在表示“反代 IP 优选配置”，不再生成本地 Clash YAML。

## 四、关键结论

- 普通测速测的是 Cloudflare 入口 IP 到 Worker 域名的连通性、TLS 和 HTTP 响应延迟。
- 普通测速不是下载测速，不测吞吐量。
- `Cf-Ray` 只用于推断入口机房和国家，国家筛选依赖内置 colo 映射。
- `stable` 模式通过多轮复测提高稳定性判断，排序更重视成功率和波动。
- 反代优选和入口 IP 测速是两条独立链路。
- 开启 `worker_verify` 后，反代测试会通过 Worker 的 `/admin/proxyip-test` 做真实出口验证，是当前更可靠的反代测试方式。
- 反代优选结果默认落地到 `PROXYIP.txt`，再推送到 Worker 动态使用。
